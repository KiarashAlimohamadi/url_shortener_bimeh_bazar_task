## 1. الگوریتم سنگین در هر Request

### توضیح سوال:
اگر پردازش الگوریتم برای هر درخواست زمان‌بر باشد و نیاز باشد داده‌ای به دیتابیس یا سرویس دیگر ارسال شود، ممکن است سرعت سرویس اصلی پایین بیاید.

### پاسخ پیشنهادی:
 - میتوان پردازش ها را مانند Source Code موجود در همین پروژه به صورت async پیاده سازی کرد.
- میتوان برای پردازش لاگ ها از message queue(RabbitMQ-KAFKA) استفاده کرد.
- استفاده از cache(Redis) .

## 2. Deploy به صورت Multi-Instance روی چند سرور

### توضیح سوال:
وقتی سیستم روی چند سرور اجرا شود، نباید چیزی وابسته به حافظه یا دیسک یک سرور خاص باشد.

### پاسخ پیشنهادی:
- زمانی که قرار است از Multi-Instance یا Distributed Systems استفاده کنیم : (کتاب system design از alex xu )
 - - استفاده از Load Balancer میتواند باعث مدیریت و کاهش ریسک پردازش زیاد روی وب سرور ها شود.
 - - درخواست ها میتوانند از طریق message queue(RabbitMQ-KAFKA) به Worker ها داده شود که هیچ request بدون response باقی نماند
 - - استفاده از دیتابیس به صورت shard های مختلف و همچنین رابطه master و slave در دیتابیس ها میتواند به سرعت کوئری زدن و افزایش دسترسی به دیتابیس بیافزاید
 - - استفاده از CDN میتواند باعث شود که وب سرور ها کمتر درگیر کارهای کوچک و کم اهمیت شوند و منابع  صرف request های مهم شوند

## 3. ترافیک سنگین (چند هزار Request در ثانیه)

### توضیح سوال:
کمپین تبلیغاتی باعث افزایش ناگهانی ترافیک می‌شود و اگر آماده نباشیم سرویس down می‌شود.

### پاسخ پیشنهادی:
- -استفاده از Load Balancer جهت تقسیم کردن ترافک ها به روی وب سرور های متفاوت به صورت مساوی
- -استفاده از cache دز هنگام ریدایرکت
- -استفاده از CDN جهت لود کردن صفحات استاتیک
